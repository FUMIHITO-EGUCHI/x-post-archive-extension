# Like Import Requirements

## Goal

X の「いいね」欄に表示される投稿を、保存時点のスナップショットとしてアーカイブへ取り込めるようにする。

この機能の目的は「いいね一覧を後から検索しやすい形で保存すること」であり、X 上のいいね状態や一覧内容を後追いで同期し続けることではない。

## Scope

初期スコープに含めるもの:

- いいね欄に表示されている投稿 1 件を手動で保存できること
- その投稿が「いいね欄経由で保存された」とタグで識別できること
- 通常タイムライン上の保存と同じ保存モデルを使えること
- viewer からタグ検索で絞り込めること
- 表示中 DOM だけでなく、いいね一覧を順次読み込んで準全件取得に近い収集ができること

初期スコープに含めないもの:

- いいね解除の追跡
- 保存済み投稿への自動反映
- 他人の返信全体や会話全体の補完取得
- X 側の内部 API 契約に強く依存した完全保証の全件取得

## User Flow

1. ユーザーが X の `Likes` 画面を開く
2. content script がいいね欄内の投稿 `article` を検出する
3. 各投稿に保存 UI を差し込む
4. ユーザー操作で対象投稿を抽出する
5. background がスナップショットとして保存する
6. `liked` 系の自動タグを付与する
7. 重複保存なら新規作成せず既存投稿を返す

全件寄りの取り込みでは次の流れを取る:

1. ユーザーが X の `Likes` 画面で一括取り込みを開始する
2. content script が現在表示中の投稿を収集する
3. 自動スクロールまたは段階スクロールで追加 DOM を読み込ませる
4. 新出 `x_post_id` を重複排除しながら収集キューへ積む
5. 停止条件に達したら収集済み投稿を順次保存する
6. 保存時に `liked` 系タグを付ける

## Technical Requirements

### 1. 画面判定

- content script は現在位置が X のいいね一覧画面かどうかを判定できること
- 画面判定は UI テキスト依存より URL と DOM 構造を優先すること
- 少なくとも `/likes` を含む URL パターンを扱えること
- 判定結果は「通常タイムライン」「詳細画面」「いいね欄」などの表示コンテキストとして内部利用できること

### 2. DOM 走査

- 投稿抽出の最小単位は既存方針どおり `article[data-testid="tweet"]` を維持すること
- いいね欄固有のレイアウト差分があっても、投稿本文・投稿 URL・投稿者・反応数・メディアを抽出できること
- 仮想スクロールや遅延描画に対応するため、`MutationObserver` ベースの再走査を継続利用できること
- 同一 `article` への重複ボタン差し込みを防ぐこと

### 3. 保存 UI の適用条件

- 保存 UI は「保存可能な投稿」にのみ出すこと
- 広告、推薦モジュール、会話コンテナ誤検出を避けること
- 投稿 ID を取得できない `article` には保存 UI を `error` 扱いで残すか、最初から差し込まないこと
- 保存済み判定は `x_post_id` をキーに既存 API を再利用すること

### 4. 抽出データ

最低限、既存保存 API に渡せる次の項目を取得できること:

- `x_post_id`
- `x_username`
- `post_text`
- `post_url`
- `posted_at`
- `reply_count`
- `repost_count`
- `like_count`
- `media[]`
- `video_candidates[]` が取れる場合は既存処理へ受け渡すこと

追加で、いいね欄由来の識別情報として次を持てること:

- 自動タグ
- 推奨タグ名: `liked`
- 必要なら補助タグとして `source/likes` のような内部運用タグを検討してよい

必要なら補助メタデータとして次を追加候補にする:

- `save_context_url`
- `save_context_label`

ただし検索価値が薄い DOM 断片は保存しないこと。

### 5. タグ付与方針

- 「いいね欄から保存した」事実は post 本体の専用フィールドではなくタグで表現すること
- 付与タグは既存タグモデルに乗せること
- 自動タグとして保持し、手動タグと区別できること
- 重複保存時も `liked` タグ未付与なら追加できること
- 既に `liked` タグがある場合は重複付与しないこと
- このタグは保存時点の文脈を示す補助情報として扱い、いいね解除の追跡には使わないこと

初期案:

- いいね欄から保存したら `liked` 自動タグを付ける
- 通常画面から保存した場合は何も付けない
- 将来ブックマーク経由も扱うなら `bookmarked` などの自動タグを追加する

### 6. 重複防止と冪等性

- 一意キーは引き続き `x_post_id` を使うこと
- いいね欄と通常画面の両方で同じ投稿が見えても二重保存しないこと
- 保存ボタン連打や DOM 再描画が起きても、DB 上の結果が安定すること
- duplicate 時も UI 側で `Saved` 状態へ遷移できること
- duplicate でも `liked` タグが未付与ならタグ付与だけは実行できること

### 7. Runtime / Background フロー

- content script は DOM 抽出だけを担当すること
- 保存要求は既存どおり background へ message で送ること
- background が入力値検証、DB 保存、メディア保存、重複処理の起点になること
- いいね欄からの保存時は background 側で `liked` 自動タグ付与まで一貫して処理できること
- 一括取り込み時は content script が収集、background が保存キュー処理を担当すること

### 8. データモデル要件

- `posts` に保存元専用フィールドは追加しない
- いいね欄由来の識別は既存 `tags` / `post_tags` で扱う
- 自動タグ付与 API は duplicate 時の再利用を考慮すること
- `liked` タグは通常のタグ検索で使えること
- 将来、保存経路履歴が必要になった時だけ専用テーブル追加を検討する

### 9. 検索 / Viewer 要件

- viewer で `liked` タグを確認できること
- 「いいね欄から保存した投稿だけ」はタグ検索またはタグフィルタで絞り込めること
- 既存タグ UX を壊さずに自動タグと手動タグを見分けられること

### 10. 権限 / 実行条件

- 拡張は X 上でのみ動作させること
- Manifest V3 前提を維持すること
- remote hosted code を使わないこと
- 追加権限は最小限にとどめること
- まずはユーザーが閲覧しているいいね欄 DOM と、その画面操作で追加読み込みされる DOM を対象にすること
- X の非公開 API や未保証 endpoint に直接依存しないこと

### 11. 準全件取得フロー

- 収集対象は「初期表示 DOM に限定しない」こと
- いいね欄上でスクロールを進めながら、追加表示された `article[data-testid="tweet"]` を継続収集できること
- 収集キーは `x_post_id` とし、同一投稿の再読込を除外できること
- 収集処理と保存処理は分離し、先に候補を集めてから保存するか、一定件数ごとにバッチ保存できること
- 停止条件を明示すること
- 停止条件候補:
- 新規投稿が一定回数連続で増えない
- ページ末尾到達を示す DOM を検出した
- ユーザーが停止した
- 安全上限件数または時間上限に達した
- 途中失敗しても、どこまで保存できたか分かる進捗を持てること
- 一括処理中にレート制限や DOM 変化が起きても、全失敗ではなく部分成功として扱えること

### 12. 一括取り込み UI 要件

- 各投稿の個別保存とは別に、いいね欄全体向けの取り込み開始 UI を持てること
- 実行中は「収集中」「保存中」「停止済み」「完了」「一部失敗」を区別して表示できること
- 収集件数、保存件数、重複件数、失敗件数を少なくとも把握できること
- ユーザーが途中停止できること
- 長時間実行を前提に、UI が消えても安全に再開しやすい構造が望ましい

### 13. 障害時の扱い

- 投稿本文抽出失敗、投稿 ID 抽出失敗、メディア保存失敗を区別できること
- 本文や投稿 ID が取れない場合は保存を中止すること
- メディア保存だけ失敗した場合の扱いは既存方針に合わせて `failed` 状態で残せること
- いいね欄固有の DOM 変化で抽出に失敗した場合、ログで切り分け可能な情報を残すこと
- 一括収集中のスクロール失敗、読み込み停止、途中中断を区別できること

### 14. 変更耐性

- X の DOM は不安定なので、抽出ロジックは `features/x/` に閉じ込めること
- いいね欄固有判定と投稿共通抽出を分けること
- 画面コンテキスト判定、投稿抽出、収集制御、保存要求生成を別関数に分離すること
- UI 文言や日本語表示ラベルに依存しすぎないこと

## Recommended Implementation Split

### Step 1: 表示コンテキスト判定を追加

- いいね欄かどうかを URL から判定する utility を追加する
- content script が現在の表示コンテキストを取得できるようにする

### Step 2: 自動タグ付与経路を追加

- いいね欄からの保存時に `liked` 自動タグを付ける処理を追加する
- duplicate 時にもタグだけ追加できるよう archive service を拡張する

### Step 3: 一括収集制御を追加

- スクロールしながら新規 `article` を収集する collector を追加する
- 停止条件、進捗、重複排除を持たせる

### Step 4: いいね欄からの一括取り込みを有効化

- 個別保存で `liked` タグを付ける
- 一括取り込みで収集済み候補を順次保存する

### Step 5: Viewer 表示を追加

- 投稿詳細または一覧で `liked` タグを確認できるようにする
- タグフィルタで `liked` を絞り込めるようにする

### Step 6: 検証

- いいね欄から未保存投稿を保存できる
- 同じ投稿を通常画面から再保存しても重複しない
- duplicate 投稿に `liked` タグだけ追加できる
- 一括取り込みで表示外の追加読み込み分も保存対象に入る
- duplicate 時の UI が壊れない
- `npm run typecheck`
- `npm run build`

## Open Questions

- 自動タグ名は `liked` で十分か、`source/likes` のような衝突しにくい命名にするか
- 一括収集は content script 主導で十分か、background に進捗状態を持たせるか
- X の無限スクロール終端をどの条件で判断するか
- 一括収集中の休止・再開を MVP に含めるか
- `/likes` 以外の類似画面でも同じ collector を使い回すか

## Recommended Decision For MVP

MVP では次を推奨する。

- 個別保存と一括取り込みの両方を対象にする
- 画面判定は URL ベース
- いいね欄由来の識別は `liked` 自動タグで扱う
- 重複時は既存レコードを返しつつ `liked` タグ不足なら補う
- 一括収集は X 画面上の段階スクロールで準全件取得を目指す
- viewer は最低限 `liked` タグの表示とタグフィルタを実装対象にする

これで「いいね欄から見つけた投稿を後で検索できる形で残す」という主目的を小さく達成できる。
