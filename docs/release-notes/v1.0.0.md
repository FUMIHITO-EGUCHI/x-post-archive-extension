# v1.0.0

Chrome Web Store 初公開リリース。v0.20.0 までで積み上げてきた連投アーカイブ機能・パフォーマンス改善・大規模アーカイブ対応に、本リリースで網羅的なセキュリティ強化を上乗せした上で、CWS の節目として 1.0.0 を切ります。

> 本リリースは内部的には v0.20.1 / v0.20.2 として準備していたものを、CWS 初公開タイミングに合わせて 1.0.0 に直接昇格させたものです。コードの実体は v0.20.0 → v1.0.0 の差分そのもので、v0.20.1 / v0.20.2 という GitHub Release は存在しません。

## ユーザーにとって良くなったこと

- **認証トークンを永続ストレージから完全排除**: X セッションの Bearer token / Transaction-ID / Client-UUID は IndexedDB に保存されなくなりました。送信時に必要な認証ヘッダーは、ブラウザを閉じれば消える session-only 領域 (`chrome.storage.session`) のみに保持されます。`x-csrf-token` (= `ct0`) は session storage にも保存せず、`cookies` permission 経由で送信時に都度取得します。フォレンジック / 端末上の他プロセス / バックアップ ZIP 経由での漏洩リスクが解消されます
- **X 上の悪意あるスクリプトからアーカイブを守る**: X ページ側で XSS が起きた場合でも、「アーカイブ全消去」「ログ削除」「タグ統合」など破壊的操作は viewer ページからしか受け付けません
- **悪意ある ZIP バックアップを安全に拒否**: 巨大 / 過剰圧縮された ZIP（いわゆる ZIP 爆弾）や、`..` 等のパストラバーサルを含むパスを早期に拒否します。エラーは `2 GB` / `50 GB` 等の human-readable な単位で表示
- **改ざんされた media を弾く**: X の画像/動画 URL から取得したレスポンスが image/video 以外の content-type だった場合、保存されず failed 扱いになります
- **viewer ページの CSP を明示宣言**: `script-src 'self'; object-src 'self'; base-uri 'self'; form-action 'self'; frame-ancestors 'none';` を Chrome Web Store の審査者にも判別可能な形で manifest に明記

## 修正内容

### セキュリティ修正

- **[High] Bearer token の永続化排除 (PRIVACY.md 整合性)**: TweetDetail GraphQL の `Authorization` / `x-client-transaction-id` / `x-client-uuid` ヘッダーを capture 時に分離。永続化される template には残さず、session-only ストレージにのみ保持。`x-csrf-token` (= `ct0`) は session storage にも持たず、`cookies` permission 経由で送信時に都度取得。`tweet_detail_template` テーブルは非機密ヘッダー (`accept-language` / `x-twitter-client-language` 等) のみ保存
- **[Medium] OPFS path 検証強化**: `requireBackupPath` と `splitOpfsPath` で `..` / `.` セグメントを reject。`buildMediaOpfsPath` / `buildVideoPreviewOpfsPath` に `xPostId` / `mediaId` の defense-in-depth validation を追加
- **[Medium] 破壊的 runtime message に sender 検証**: `archive/reset` / `logs/clear` / `posts/delete` / `tag.rename` / `tag.merge` / `tag.redirects.delete` / `tag.bulk-assign.apply-batch` / `refetch.clear` / `post_tag.remove` は extension viewer page (`chrome-extension://<id>/...`) からのみ受理
- **[Medium] runtime message payload の structural validator**: 全 message type に対し `xPostId` フォーマット (`/^\d{1,30}$/`)、tag 名 200 字制限、batch 5000 件上限、`tweet-detail-template/set` の URL は https x.com / twitter.com のみを強制 (互換性のため両 canonical を許可)
- **[Medium] MAIN→ISOLATED CustomEvent の handshake**: ISOLATED 起動時に `crypto.randomUUID()` で token 生成 → `postMessage` で MAIN へ伝達 → MAIN は capture 時に CustomEvent detail に token を含めて dispatch → ISOLATED は timing-safe 比較で一致確認。X 上の悪意ある script による template spoof を阻止
- **[Medium] ZIP bomb 防御**: import 時に entry 数 200,000 / 個別 2 GB / 合計 50 GB / manifest 256 MB の pre-flight 上限を導入。サイズ判定は ZIP central directory の `uncompressedSize` メタデータに依存し、攻撃者が実展開サイズと申告値を意図的にずらしうるが、本拡張の脅威モデル (個人バックアップの誤爆防止) では許容範囲
- **[Low] OPFS path 構築側で xPostId / mediaId の正規表現検証** (defense-in-depth)
- **[Low] viewer 拡張ページに CSP 明示宣言**
- **[Low] media fetch の content-type allowlist** (image/* または video/* のみ)

## 詳細

### Bearer token をどこに置くか

Chrome Web Store 審査と PRIVACY.md の整合のため、X セッション認証ヘッダーは disk-backed ストレージ (IndexedDB) には書かず、`chrome.storage.session` (browser を閉じれば消える ephemeral 領域) にのみ保持する設計に変更しました。

副作用として、ブラウザを完全終了 → 再起動した直後にスレッド refetch を走らせる場合、X タブを一度開いて自動再 capture されるまで `template-missing` エラーが返ります。X タブを開けば即座に自動復帰します。

### 破壊的 message を viewer 限定にした理由

`chrome.runtime.onMessage` のリスナーは extension の content script (X 上で動作) からも extension viewer page からも区別なく呼ばれます。これまでは sender の出自を検証していなかったため、X 上で XSS が起きた場合に content script を経由して `archive/reset` を発火させ、ユーザーのローカルアーカイブを全消去させる経路が理論上存在しました。

このリリースから、破壊的操作は `sender.url?.startsWith(chrome.runtime.getURL(""))` を満たすメッセージのみ受理します。content script から送れるのは additive な操作 (`posts/save`, `posts/has` 等) のみです。

### handshake token は完全防御ではない

X 上で攻撃者が任意の JavaScript を実行できる前提では、ISOLATED → MAIN へ送る handshake `postMessage` を盗聴できる余地があります（同一 window 上のすべてのスクリプトが `message` イベントを listen 可能）。本リリースの handshake は、token 不一致の event を確実に拒否することで、X SPA 内部や偶発的な競合スクリプトからの誤動作を防ぎます。完全な spoof 阻止のためには将来 `chrome.scripting.executeScript({ world: "MAIN", args: [token] })` での動的注入に切り替える必要がありますが、今回は patch スコープに留めました。

### 1.0.0 を切る意図

CWS への初公開を 1.0.0 のマイルストーンとして扱います。本リリース以降の DB schema 変更や互換性破壊は引き続き SemVer に従い、breaking change は major bump (2.0.0) で扱う想定です。CWS 上のユーザーから見たバージョン番号はストアページにのみ現れるため、内部開発のテンポを縛ることはしません。
