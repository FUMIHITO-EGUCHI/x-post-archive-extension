# v0.20.0

## ユーザーにとって良くなったこと

- **連投（スレッド）保存**: X の self-reply チェーンを OP 投稿者本人だけ抽出して保存・閲覧できるようになりました。スレッドカードはルート1件に集約し、クリックで全コマを縦に展開します
- **likes import 連動**: 通常保存中や likes import 中も、保存対象がスレッドの一部だった場合に続編を自動で取りに行きます
- **大規模アーカイブの高速化**: 17,000 件超のアーカイブで、ページ送り（offset 5,000 相当）の取得が約 110 ms → 約 2 ms に短縮されました（約 50× 改善）
- **スレッドカードの件数表示**: ルート投稿カードに「全 N 件」バッジが付き、スレッド単位の規模が一目で分かります
- **TweetDetail 認証切れの通知**: スレッド取得に必要な認証情報が古くなったとき、viewer 上で再キャプチャを促す通知を表示するようになりました
- **viewer リンクの安全化**: 保存済み投稿の URL に `javascript:` / `data:` スキームが含まれていてもリンクとして発火しないよう防御を追加しました

## 詳細

### スレッドアーカイブの設計判断

spike 段階で「隠れタブで DOM を読み込んで抽出」案も検討しましたが、X 側の負荷とブラウザリソース消費が大きく不採用としました。最終的に TweetDetail GraphQL を直接叩く方式を採用し、ユーザー操作起点で 1 リクエスト / 5 秒に throttle することで rate limit を踏まないよう設計しています。詳細は `docs/findings/2026-04-26-thread-archive-spike.md` を参照。

### keyset pagination が必要だった理由

IndexedDB の `cursor.advance(N)` は文字通り「N 回前進」する O(N) 操作です。アーカイブが 17k 件を超えたあたりで、ビューアの「もっと読み込む」が体感できる遅延として顕在化していました。複合インデックス `[saved_at+x_post_id]` を使えば、前ページ末尾の `(value, x_post_id)` を境界として `IDBKeyRange.upperBound([v, id], true)` で次ページに直接ジャンプでき、深さに依存しない O(log N) になります。これは `reply_count` などの非ユニーク値を含むソートでも、`x_post_id` のタイブレークで安定したページングが取れます。

### TweetDetail template stale 通知

X が再ログインや UI 改修で `Authorization` トークンを更新すると、保存済みのテンプレが使えなくなり、スレッド展開が失敗していました。これまでは worker のリトライが上限に達するまで気付けませんでしたが、今回 `runtime.onMessage` で auth-stale カウントを返す API を追加し、viewer 上で「X を開いて再キャプチャ」を促すバナーを出すようにしました。
