# Future Features Memo

## Purpose

MVP 完了後に検討したい機能のメモを残す。
ここに書く内容は「今すぐ実装する項目」ではなく、優先度や前提条件を整理しながら将来の判断材料として扱う。

## Priority Order

判断に迷ったときは、次の順で優先する。

1. 検索性を上げる機能
2. 保存時点の文脈を保ちやすくする機能
3. 保存データの扱いやすさを上げる機能
4. バックアップや運用補助
5. 見た目だけを改善する機能

## Candidate Features

### 1. タグ運用とタグ起点の表示制御

- ハッシュタグ自動タグ化
  - 投稿本文からハッシュタグを抽出して検索キーとして扱う
- 保存経路の自動タグ付け
  - いいね経由で保存した投稿に自動タグを付けるオプションを用意する
  - ブックマーク経由で保存した投稿に自動タグを付けるオプションを用意する
- 手動タグ付与
  - あとから自分の観点で整理できるようにする
- タグホワイトリスト
  - 初期画面で表示対象に含めるタグ群を指定できるようにする
- タグブラックリスト
  - 初期画面で非表示にしたいタグ群を指定できるようにする
- 表示モードの排他選択
  - ホワイトリスト方式かブラックリスト方式のどちらか一方だけを選べる仕様にする
- タグによる表示制御プリセット
  - ホワイトリスト・ブラックリスト・並び順などの表示条件を保存し、後で呼び出せるようにする
- タグ一覧
  - 利用中タグ、件数、未使用タグを見渡せるようにする
- タグ管理画面
  - 使っていないタグや表記ゆれを整理できるようにする

期待する価値:
- 保存件数が増えても、自分用の観点で素早く見たい集合を呼び出せる

### 2. 本文検索の強化

- 複数条件検索
  - 投稿者、本文、タグ、保存日を組み合わせて絞り込めるようにする
- ソート改善
  - 保存日順だけでなく、関連度や投稿日時でも並べ替えられるようにする
- ハイライト表示
  - 検索語が本文のどこにあるかを一覧と詳細で見つけやすくする
- 事前インデックス作成
  - 将来の検索高速化のために、時間のかかるインデックス作成を事前実行できるようにする
  - 保存済みデータ全体を対象に、検索用の補助インデックスをローカルで構築できるようにする

期待する価値:
- 保存件数が増えても探しやすさを維持しやすい

### 3. オプションとストレージ管理

- オプション画面
  - 表示制御、保存方針、バックアップ関連の設定をまとめて管理できるようにする
- 自動タグ付けオプション
  - いいね保存時の自動タグ付与を個別に ON/OFF できるようにする
  - ブックマーク保存時の自動タグ付与を個別に ON/OFF できるようにする
- 事前インデックス作成オプション
  - 時間のかかる検索用インデックス構築を手動で実行できるようにする
  - 自動実行ではなく、ユーザーが必要なタイミングで走らせられるようにする
- ストレージ利用量表示
  - 利用可能容量と現在使用中の容量を viewer または options で確認できるようにする
- 保存先ごとの内訳表示
  - IndexedDB と OPFS のどちらがどれだけ使っているか把握しやすくする

期待する価値:
- ローカルアーカイブとしての運用状態を自分で管理しやすくなる

### 4. スレッド保存

- 投稿者本人の返信連投チェーン保存
  - このプロジェクトで定義しているスレッド本体を保存対象にする
- スレッド順表示
  - 詳細画面で連投順が崩れないように表示する
- 単体投稿とスレッドの両対応
  - 保存単位は投稿のまま維持しつつ、表示ではまとまりを扱えるようにする

期待する価値:
- 単発投稿だけでなく、文脈付きで読み返しやすくなる

### 5. 保存時点メタデータの拡張

- 投稿日時の保存
  - 保存時点で見えていた投稿日時を保持し、詳細画面で表示できるようにする
- 反応数スナップショットの保存
  - 保存時点の返信数、いいね数、リポスト数を保持する
- 詳細表示への反映
  - 本文やメディアとあわせて、保存時点の数値として見返せるようにする
- 検索や並び替えへの活用
  - 将来的に投稿日時や反応数を検索条件やソート条件へ使えるようにする

期待する価値:
- 保存時点の文脈を、本文以外の情報も含めて見返しやすくなる

### 6. メディア対応の拡張

- 動画保存
  - MVP 後の大きな拡張候補として扱う
- 画像・動画メタデータの拡張
  - 長さ、サムネイル、解像度など、後から見返すための情報を増やす
- メディア個別ダウンロード
  - 保存済みの画像や動画を、viewer から個別にダウンロードできるようにする
  - バックアップとは別に、必要なメディアだけ書き出せるようにする
- ローカル保存最適化
  - 保存容量と再生体験のバランスを取れるようにする

期待する価値:
- スクリーンショットでは代替できない投稿の再現性を上げられる

#### X 動画ダウンロードの技術要件メモ

前提:
- X 側の配信仕様は固定ではないため、動画取得方式は「現在の DOM / Network 観測に依存する実装」であることを明記する
- 初期実装は DRM 付き配信や保護コンテンツを対象外とし、通常の取得可能な動画だけを扱う
- 本機能は MVP に含めず、保存済みアーカイブの再現性を上げる後続機能として扱う

取得対象の整理:
- 投稿に紐づく動画メディアを画像メディアとは別種別で管理する
- 単一 mp4 URL だけでなく、HLS の master playlist / media playlist / segment 群を扱える前提にする
- 複数品質がある場合は、最高画質固定ではなく「既定品質 + 手動選択」に拡張できる設計にする
- 動画が取得できない場合でも、サムネイル、再生時間、失敗理由などのメタデータは残せるようにする

抽出要件:
- content script だけで完結させず、X 画面上で得られる動画 URL・poster・variant 情報を抽出する責務と、実ダウンロード責務を分離する
- DOM 直読みだけに依存せず、必要なら performance entries や XHR / fetch 監視など取得経路を追加検討できる構造にする
- X 固有のセレクタや URL 判定は `features/x/` 配下に隔離し、viewer や DB 層に漏らさない
- 抽出値はそのまま永続化せず、アプリ内部型へ変換してから保存する

ダウンロード要件:
- 小さい mp4 直リンクなら `fetch` で Blob 化して OPFS に保存できる
- HLS の場合は playlist 解析、segment の逐次取得、必要なら結合処理を行うパイプラインが必要になる
- ブラウザ内だけで結合する場合は、TS / fMP4 の扱い、メモリ使用量、長尺動画の失敗率を先に評価する
- 初期段階では「ローカル再生用に segment 群と manifest を保存する案」と「単一 mp4 に変換して保存する案」を分けて比較する
- サービスワーカーは DOM を持たないため、Blob URL や変換処理が必要なら offscreen document の採用可否を検討する

保存要件:
- メインの検索用メタデータは IndexedDB、動画本体や segment は OPFS に保存する
- 動画 1 件に対して、論理メディア ID、親投稿 ID、取得方式、品質、mime type、duration、width、height、thumbnail 情報を持てるようにする
- HLS 保存を許容するなら、1 動画 = 1 ファイル前提にせず、playlist ファイルと複数 segment を束として管理できるようにする
- 保存状態は `pending`, `downloading`, `ready`, `partial`, `failed` などを持ち、途中失敗を区別できるようにする
- 再試行時は重複保存を避けるため、投稿 ID だけでなく動画 variant 単位の一意性を持たせる

再生・閲覧要件:
- viewer は外部 CDN URL に依存せず、保存済み OPFS データから再生できることを基本方針にする
- 単一 mp4 なら object URL 再生、HLS 保存ならローカル manifest を読み出す再生方式を検討する
- 取得失敗時や一部欠損時には、再生不能であることと失敗理由が分かる UI を出す
- 長時間動画や高解像度動画で viewer が固まらないように、一覧ではサムネイル中心、詳細でのみ再生を開始する

権限・実行環境要件:
- Manifest V3 前提で、必要最小限の `host_permissions` を X 関連ドメインと動画配信ドメインに限定する
- ユーザー向けの書き出し機能を付ける場合は `downloads` 権限を追加する
- DOM を使う変換や Blob URL 生成をバックグラウンドで扱うなら `offscreen` 権限を追加候補にする
- remote hosted code は使わず、playlist 解析や必要な変換ロジックは拡張内に同梱する

容量・失敗時要件:
- 動画保存前に概算サイズが分かる場合は容量見積りを出し、分からない場合も保存中サイズを更新する
- `navigator.storage.estimate()` を使った空き容量の目安表示を前提にする
- 保存中断、通信失敗、segment 欠損、権限不足を別エラーとして扱う
- 途中まで保存できた場合はロールバックか partial 保持かを明示的に決める
- 容量圧迫を避けるため、動画保存は投稿保存とは別オプションにし、既定で自動保存しない方針を優先する

実装前に切るべき検証項目:
- X 上の動画が実際に mp4 直リンク中心か、HLS 中心か
- HLS の playlist / segment を拡張コンテキストから安定取得できるか
- OPFS 保存後のローカル再生が Chrome 拡張ページ上でどこまで素直に動くか
- 長尺動画でメモリ使用量と保存時間がどの程度になるか
- viewer での再生を優先するか、ユーザーのローカルファイル書き出しを優先するか

暫定調査結果:
- `samples/video_samples.html` では `video` 要素の `src` が `https://video.twimg.com/tweet_video/...mp4` になっており、このサンプルは mp4 直リンク型だった
- 一方で、公開されているダウンロード実装の中には、`.m3u8` playlist と `.m4s` segment を取得して mp4 化するものもあり、少なくとも一部の動画系面では HLS 経路が存在する
- 公開ダウンロードサービスは概ね「複数 quality の mp4 を返す」UI を出しており、ユーザー向けの最終成果物は mp4 だが、その裏側が常に mp4 直取得とは限らない
- したがって初期実装方針としては「公開ポスト本文中の通常動画は mp4 variant の抽出を第一候補」「HLS は取りこぼし回避の第二経路」として設計するのが妥当

### 7. 文脈保存の補助

- 関連返信の選択保存
  - スレッド本体とは別概念として、有用な他者返信を任意で残せるようにする
- スレッド補完取得
  - 起点投稿から不足分を追加取得する導線を将来検討する

期待する価値:
- 全会話保存に踏み込まずに、必要十分な文脈を残しやすくする

### 8. バックアップと持ち運び

- エクスポート
  - 投稿、タグ、表示制御プリセット、メディア本体をまとめたパッケージ形式で外部保存できるようにする
- インポート
  - パッケージを読み込んで別環境へ復元できるようにする
- 整合性チェック
  - IndexedDB と OPFS の欠損や孤立データを確認できるようにする
- オフライン完結
  - クラウド同期を前提にせず、ローカルファイルの入出力だけで完結できるようにする

期待する価値:
- 個人アーカイブとして長く使いやすくなる

### 9. 完全オフライン運用の強化

- 保存済みビューの完全オフライン動作
  - 一覧、詳細、検索、タグ表示制御を通信なしで完結させる
- オフライン前提の UI 設計
  - 外部 API や CDN に依存しない構成を維持する
- 復旧導線の明確化
  - インポート、整合性チェック、容量確認をローカルで回せるようにする

期待する価値:
- サービス変更や通信状態に影響されにくい個人アーカイブになる

## Not For Near Term

次は候補に見えても、近い段階では優先しない。

- X UI の完全再現
- 会話全体の完全保存
- 保存済み投稿の自動更新追従
- 複雑な自動分類
- 推薦機能
- 共有前提の SNS 的機能

- スレッド補完取得は手動操作にするか、保存時オプションにするか
